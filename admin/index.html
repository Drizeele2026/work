这个工具怎么工作

它是一个零后端的值班排班页面。公开成员打开页面直接看排班，负责人本地配置好 token 后，再把排班数据写回 GitHub 仓库里的文件。

数据放在哪里

  • 页面文件是 index.html。
  • 组织索引放在 data/organizations.json，当前组织排班放在 data/orgs/{slug}/schedule.json。
  • 介绍内容可以放在 data/about.md，后续再接页面渲染。
  • 仓库地址是 Drizeele2026/work。

排班规则

每个团队独立排班，按成员名单顺序循环。负责人只维护团队名称和成员名单，系统会按已发布规则继续顺排。

排班按自然日连续推进，周一到周日都算一天，不跳过周六周日。查看后续月份时，会沿用当前规则继续往后推，不会每个月重新从第一个人开始。

为什么这样排

核心目标是公平，而且规则要透明。顺序轮转让每个人都按同一条队列往前走，人数不能整除月份天数时，最多也只是相差 1 天。

周六周日也放在同一条自然日队列里，周末值班会随着日期一起往后滚。这样大家都有份，不会固定压在几个人身上，也不会有人长期避开周末。

跨月继续接龙也是同一个原因。如果每个月都从第一个人重新开始，月初、月末和周末容易反复落到同一批人身上。连续推算可以把这种偏差摊开。

系统实现原理

这个工具不需要服务器。公开页直接读取仓库里的排班文件，团队成员打开链接就能看，不需要登录，也不需要配置 token。

需要改排班时，只有负责人进入管理地址，在可信电脑上配置一次 GitHub PAT。点击发布时，本质上就是把当前组织的排班文件写回 data/orgs/{slug}/schedule.json。

GitHub Pages 负责托管公开页和管理页；GitHub Actions 负责跑自动提醒；提醒规则写在 .github/workflows/duty-reminder.yml 这个 workflow 里。

所以仓库就是持久化存储，commit 就是保存记录。每次修改都有提交历史，出了问题可以对比、追溯，也可以回滚。

当前版本

这版把普通查看和管理页面分开。团队成员看排班和复制内容，负责人使用单独管理地址改成员、重排后续月份和保存。
