# 排班模型瘦身设计

## 背景

当前管理页把几个概念混在一起了：

- 成员名单顺序。
- 接龙节点。
- 兜底起点。
- 生成本月排班。
- 当天值班人 / 前一天值班人。

这些概念能处理“临时调班”“从某天重新接上真实排期”这类复杂场景，但现在明确不做这些。私下换班由同事自己处理，不进入系统。

这个工具只需要解决一件事：按已发布的团队成员顺序，稳定算出某天每个团队轮到谁。

## 目标

- 管理员只维护团队名称、成员顺序、成员 OpenID。
- 发布后，从发布当天开始使用新规则。
- 发布当天之前的排班不主动改变。
- 新团队没有历史时，第一天从成员名单第一个人开始。
- 不再要求管理员理解接龙节点、兜底起点、月份生成。
- 前端、后端、测试使用同一套规则。

## 非目标

- 不支持临时调班记录。
- 不支持手动从某天重新接上。
- 不支持管理员手动添加生效节点。
- 不再把“生成某月排班”作为核心流程。

## 推荐方案

采用最干净的数据模型：发布版本。

不再使用 `anchors` 和 `last` 作为正式数据结构。系统保存一组按日期生效的规则版本，每个版本包含当时完整的团队配置。

示例：

```json
{
  "version": 2,
  "updatedAt": "2026-07-04T10:02:52.436Z",
  "current": {
    "teams": [
      {
        "name": "测试",
        "color": "violet",
        "members": [
          { "name": "许绵绵", "feishuOpenId": "ou_x" },
          { "name": "郑成清", "feishuOpenId": "ou_y" },
          { "name": "谭贤", "feishuOpenId": "ou_z" }
        ]
      }
    ]
  },
  "ruleVersions": [
    {
      "effectiveDate": "2026-07-01",
      "teams": [
        {
          "name": "测试",
          "color": "violet",
          "members": [
            { "name": "许绵绵", "feishuOpenId": "ou_x" },
            { "name": "郑成清", "feishuOpenId": "ou_y" },
            { "name": "谭贤", "feishuOpenId": "ou_z" }
          ]
        }
      ]
    },
    {
      "effectiveDate": "2026-07-04",
      "teams": [
        {
          "name": "测试",
          "color": "violet",
          "members": [
            { "name": "许绵绵", "feishuOpenId": "ou_x" },
            { "name": "郑成清", "feishuOpenId": "ou_y" },
            { "name": "谭贤", "feishuOpenId": "ou_z" },
            { "name": "陈鸿历", "feishuOpenId": "ou_new" }
          ]
        }
      ]
    }
  ]
}
```

`current.teams` 是管理页编辑的当前名单。`ruleVersions` 是已发布历史，用来保证历史不乱。

发布时只有内容变化才追加新版本。重复点击发布但名单没变，不追加版本。

## 排班计算规则

计算某天 `D` 的值班人：

1. 找到 `effectiveDate <= D` 的最新规则版本。
2. 对每个团队取该版本里的成员顺序。
3. 找到这个版本生效当天的起算人。
4. 从起算人开始，按成员顺序每天往后轮。

起算人的规则：

- 如果这是团队第一次出现，起算人是成员名单第一个人。
- 如果之前已有规则版本，就先用旧规则算出生效当天原本轮到谁。
- 如果这个人仍在新名单里，就从这个人开始接着排。
- 如果这个人已经不在新名单里，就从新名单第一个人开始。

判断“是不是同一个人”时优先用 `feishuOpenId`，没有 OpenID 时再用姓名。这样只改姓名时，不会误认为是两个人。

这样不需要手动 `anchors`，也不需要 `last`。发布当天的“接上”由系统根据上一版规则自动推导。

例子：

- 2026-07-04 发布测试组新名单。
- 发布前 2026-07-04 测试轮到郑成清。
- 新名单里还有郑成清。
- 所以 2026-07-04 仍是郑成清，2026-07-05 按新名单顺序继续往后排。

## 管理页变化

保留：

- 团队名称。
- 成员名单。
- 成员 OpenID 状态。
- 发布到公开页。
- 当前月份预览。

删除：

- 兜底起点。
- 接龙节点区域。
- 添加节点按钮。
- 当天值班人 / 前一天值班人选择。
- “自动生成本月排班”文案。

文案调整：

- “修改值班名单”改成“维护值班规则”。
- “自动生成本月排班”改成“按已发布规则预览”。
- “发布到公开页”表示保存当前规则版本，并刷新公开页。

## 数据迁移

现有数据里还有 `config.teams[].last`、`config.teams[].anchors` 和 `months`。

迁移策略：

1. 读取现有数据，先保证当前公开页显示结果不变。
2. 把当前团队配置迁到 `current.teams`，去掉 `last` 和 `anchors`。
3. 把现有的有效规则转换成 `ruleVersions`。
4. 迁移后新逻辑只读 `ruleVersions` 和 `current.teams`。
5. `months` 不再作为长期数据源；如果保留，只作为旧数据兼容输入，不再继续写新的月份快照。

当前 2026-07-04 的测试组变更，需要保留为一个 2026-07-04 生效的规则版本。迁移后仍应满足：

```text
2026-07-04 测试 郑成清
2026-07-05 测试 谭贤
2026-07-06 测试 钟右梅
2026-07-07 测试 陈鸿历
```

## 提醒脚本和公开页

公开页、管理页预览、飞书提醒都调用同一套排班计算函数。

它们不再依赖某个月是否已经生成快照。传入任意日期，都按 `ruleVersions` 推出当天排班。

如果没有任何规则版本：

- 有团队成员时，从名单第一个人开始。
- 没有团队成员时，提示“请先维护团队成员并发布”。

## 测试范围

需要覆盖这些用例：

- 新团队第一次发布，从第一个成员开始。
- 只新增测试成员，前端和后端顺序不变。
- 发布当天原值班人仍在新名单里，当天不变，后续按新顺序排。
- 发布当天原值班人被移除，从新名单第一个人开始。
- 历史日期使用当时的规则版本，不被后续发布影响。
- 管理页不再出现兜底起点、添加节点、当天值班人、前一天值班人。
- 公开页和提醒脚本对同一天算出的值班人一致。

## 风险

- 这是一次数据模型切换，改动比隐藏 UI 更大。
- 需要认真迁移当前 `schedule.json`，否则会影响 2026-07-04 前后的排班。
- 旧文档和旧校验脚本里大量写着接龙节点，需要同步删除或改名。

这个风险是值得的。因为继续保留 `anchors/last/months`，以后每次改名单都会绕回“到底该填哪个起点”的问题。
